## Decisions made

### 1. Operating-model reset — the framing change behind everything in this session

This session began as the bridge-pick decision called for in the v2 spec.
Ibby pushed back: the project is a research lab, not a one-strategy pipeline,
and the validation gate calibration was being applied to every iteration
when it should only apply at the live-capital transition. Three personas
agreed; June 30 deadline dropped; sessions 34–38 reframed.

### 2. Function-style strategy modules instead of template-style

`vwap_reversion_v1.py` uses a class-based template registration pattern
that supports regime filters, Mulligan, and other complex extensions.
`zn_macd_pullback.py` uses a simpler `generate_signals(df, **params)`
function pattern that the CLI `backtest` command imports directly. The
two new FX strategies use the function pattern. Reasons: simpler to read,
matches what the CLI expects, no need for the template machinery yet —
the YAML-only authoring (session 36) is where the template approach
returns. No regression.

### 3. Stale 6A 2024-01-01 demo files were shadowing the full 16-year files in `rebuild_features`

The rebuild_features step does `sorted(glob('{symbol}_1m_backadjusted_*.parquet'))[-1]`.
Alphabetical sort puts `_2024-01-01_2024-01-31.parquet` after
`_2010-01-01_2026-05-03.parquet` (because "2024" > "2010" lexically).
The session 28 demo files (only 2024-01 of 6A) shadowed the full file we
just downloaded, so the first features rebuild only covered Jan 2024.
6C didn't have this issue because session 28 didn't pre-build a 6C clean.

Resolution: moved the stale demo files to `data/_archived_session28_demo/`
and re-ran features for 6A. This is a workaround. The proper fix
(`rebuild_features` should sort by date-range coverage, not by filename)
is **observed debt** logged below — out of scope for this session.

### 4. Both strategies lose money at default knobs — and that is fine in this session's frame

Strategy 1 (6A VWAP MR + ADX): Calmar −0.04, profit factor 0.57, win rate 39.4%.
Strategy 2 (6C Donchian BO + EMA): Calmar −0.04, profit factor 0.71, win rate 32.8%.

Under the v2 framing this would be a sprint-33-style FAIL. Under the
reset operating model these are exploration runs that produce
**information about where the parameter space and feature engineering
need to go.** The mentor's design doc explicitly anticipated this:
> "These are honest first attempts. They might work. They might not.
> The point of having a research lab is finding out cheaply."

What we will do with these results in **session 35 onward**:

- **Sweep parameters.** Both strategies have parameters (entry threshold,
  stop multiple, ADX cutoff, breakout period) that are at default values.
  Once the sweep tool exists in 35, we run grids and look for regions
  where the strategy works — or confirm there's no edge to find.
- **Look at the trade logs on the chart.** The replay app will show
  where the trades happen. If 6A's losses cluster during sustained trends
  (i.e. ADX filter is too lagging), we know what to fix. If 6C's losses
  are mostly false breakouts during ranging-but-EMA-aligned periods,
  we know to tighten the trend filter.
- **Honest about what we may learn:** maybe 6A doesn't mean-revert
  reliably enough at 15m to be tradeable. Maybe 6C's breakouts fail
  faster than 3× ATR can recover. If the lab tells us that, that is
  a real finding and we redirect, no shame in it.

## Observed debt (out of scope; logged for future sessions)

1. **`rebuild_features` sort logic.** `sorted(glob(...))[-1]` picks the
   alphabetically last file, not the file with the widest date range.
   Stale demo files can shadow full files. Fix: sort by date-range span
   parsed from filename, OR pin the canonical CLEAN file via an explicit
   manifest pointer. Suggested target: session 35 or earlier when
   convenient.

2. **Trial registry has no `mode` field yet.** Plan v3 calls for
   `mode: "exploration" | "validation"` and `parent_sweep_id` columns to
   be added in session 35. The two new trials in this session are
   tagged "exploration" only in the work log, not in the registry itself.

3. **Backtest summary reports `Max drawdown (%)` of `-6468%` and `-4986%`.**
   That percentage formula clearly normalises against an initial-equity
   number that is too small (or zero-base) on losing strategies. The
   absolute USD drawdown ($146k for 6A, $80k for 6C) is the right number
   to read. The percentage normalisation is a known-cosmetic display bug
   that does not affect the underlying math. Worth fixing in a future
   summary-report session.

4. **One pre-existing test skip remains** (SHAP/numba JIT crash on Windows).
   Pre-existing, not caused by this session.
