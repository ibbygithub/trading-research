## Decisions made

- **Rule M-2 corrected (Ibby, session 32):** Gate is a floor for long entries — `new_entry >= orig.entry - N×ATR` — meaning scale-in CAN go deeper than original entry but only within N×ATR. Default N=0.3 is intentionally loose; session 33 parameter sweep should include `{0.3, 0.5, 1.0, 1.5, 2.0}`.
- **Distinct trade records for Mulligan leg:** Two calls to `_close_trade` at position exit — original leg uses `orig_qty_for_first_leg`, Mulligan leg uses `mulligan_qty`. Both share same exit_ts/exit_price/exit_reason.
- **Combined target from avg entry, not VWAP:** `combined_target = combined_avg_entry ± mulligan_target_atr × ATR`. This allows the target to be parameterized and swept independently from the original VWAP-relative target. Both are separate knobs.
- **Combined dollar risk logged, not blocked:** Ibby's 0.1→0.3 example means combined dollar risk can exceed original per-trade limit. Risk is logged at scale-in time; `max_scale_ins=1` is the primary size limiter.
- **Scale-in size defaults to `BacktestConfig.quantity`** (not re-calling `size_position`). Dynamic Mulligan sizing is future work (session 33).
- **Mulligan only fires when `exit_rules` returns `scale_in`:** `VWAPReversionV1.exit_rules` always returns `hold`, so Mulligan never fires on existing strategies. The infrastructure is wired; activation requires strategy logic to return `scale_in`.
- **Regime filter required in Mulligan config:** Data scientist recommendation. Without a regime filter, VWAP deviation signals in trending markets can look "fresh" but aren't truly independent. The Mulligan YAML includes `volatility-regime` filter.
