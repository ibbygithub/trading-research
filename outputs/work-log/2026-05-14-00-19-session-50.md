## Decisions made

- **Pre-flight halt before doing session-50 work.** Sessions 48 and 49
  sat on unmerged branches; develop was missing chapters 31-44 and 49
  plus the CLI work this session cross-references. Stopped, surfaced
  to operator, got authorization to merge 48 → develop → push, then
  branch 50 off develop. Cleanest history.
- **Run IDs only for pipeline-driving CLIs.** `verify`, `inventory`,
  `status`, `validate-strategy`, `migrate-trials`, and `tail-log` are
  read-only or one-shot; allocating a run_id and writing a JSONL file
  per invocation produces low-value clutter. The seven CLIs that
  produce artifacts (`pipeline`, `rebuild clean`, `rebuild features`,
  `backtest`, `walkforward`, `sweep`, `report`) generate run_ids.
- **Skip file logging under pytest.** `_init_cli_logging` checks
  `PYTEST_CURRENT_TEST in os.environ` and short-circuits the file
  handler attach. Without this, the existing CLI test suite would
  produce `logs/` directories in the project root every test run.
  The check is harmless in production.
- **ProcessorFormatter pattern over single-renderer structlog.** The
  pre-session config rendered ConsoleRenderer in the structlog chain
  and routed to stdlib logging as a pre-formatted string. That worked
  for stderr but ruled out a per-handler renderer (JSONRenderer for
  file). The new config uses `ProcessorFormatter.wrap_for_formatter`
  so a single contextvars-bound event_dict reaches both handlers, each
  applying its own renderer. Cost: more setup; benefit: file and
  stderr can diverge cleanly.
- **`_DynamicStderr` stream proxy.** The first time my new tests ran
  in the full suite, `tests/eval/test_trial_registry_versioning.py::
  test_dsr_across_cohorts_returns_none` failed because pytest's
  `capsys` replaces `sys.stderr` at Python level *after* my
  `StreamHandler` was constructed. The handler still wrote to the
  original stderr reference, bypassing `capsys`. The proxy re-resolves
  `sys.stderr` on every `write()`, restoring `capsys` visibility.
  This is a stronger fix than gating my handler on a test env var
  because it preserves behaviour for any other capsys-using test.
- **Schema fields bound via contextvars, not function arguments.**
  The hot-path modules' existing log calls (`logger.info("event",
  key=value)`) needed no rewrite. Once the CLI binds
  `run_id`/`stage`/`symbol` into contextvars, every downstream call
  inherits them. The only mass-editing this session did was adding
  stage-boundary logs where none existed and tagging some new events
  with `action`/`outcome`. Existing calls that pre-date this session
  still emit valid events — they just carry fewer schema fields when
  no contextvars are bound, which is the correct behaviour for
  test-context logging.
- **Stripped `get_logger` from main.py imports.** Initially imported
  it alongside the three helpers, then noticed it wasn't used in
  main.py itself (only the helper which doesn't log). Removed to
  silence the F401.
- **Pre-existing ruff violations in `cli/main.py` swept by auto-fix.**
  Running `ruff check --fix` on main.py touched lines I had not edited
  (Optional → `| None`, etc.). The session-47/49 convention is "no
  NEW violations introduced; pre-existing not in scope," and the
  auto-fixes did not regress any tests. Decided to keep the fixes —
  reverting them would be more invasive than the original sweep.
