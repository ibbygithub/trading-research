## 56.5.0 What this chapter covers

In 30 days the project went from roughly 1.5 GB to ~3.7 GB on disk.
The pressure is gigabytes, not file count: the project holds about
1,700 files outside the Python toolchain, a number that grows
predictably with each new instrument and each backtest. What grows
*unboundedly* without intervention is two things: per-strategy run
directories under `runs/` (one strategy already holds 153 sub-files
from sweep variants, and sweeps produce one timestamped directory per
variant) and the cumulative disk weight of date-stamped CLEAN and
FEATURES parquets that are not auto-reaped after a refresh. The
platform has no automated cleanup. That trajectory is unsustainable:
adding two more instruments and running a normal sweep cadence for
one quarter puts the project past 10 GB with no mechanism to reclaim
space.

This chapter does three things:

1. **Quantifies what is actually on disk today and where it goes** —
   per-layer footprint, per-instrument cost, per-category growth rate.
2. **Specifies the cleanup CLI** — five subcommands under
   `trading-research clean`, with full grammar, default behaviour, and
   manifest-aware safety rules.
3. **Defines the default retention policy** — what is reaped, what is
   archived, what is kept forever.

After reading this chapter you will know exactly what the platform
will (eventually) do when you type `clean`, what it will refuse to do,
and where the policy levers are.

This chapter is roughly 6 pages. It is referenced by Chapter 4 (Data
Pipeline §4.12), Chapter 32 (Trial Registry), Chapter 49.16 (the
`status` CLI), and the cold-start procedure in Chapter 54.

## 56.5.1 Per-instrument disk footprint

A fully built instrument — 16 years of history, both back-adjusted and
unadjusted 1m, all higher-timeframe resamples (5m, 15m, 60m, 240m,
1D), and the base-v1 feature stack — costs roughly **900 MB to 1 GB**
on disk. The detailed per-layer breakdown is in
[Chapter 4 §4.12.2](04-data-pipeline.md#4122-per-instrument-cost-breakdown).

Observed footprint (May 2026, four instruments with varying
coverage):

| Directory | Size | File count | Notes |
|-----------|------|------------|-------|
| `data/raw/` | 737 MB | 354 | RAW contracts + manifest sidecars + the original ZN bulk pull |
| `data/clean/` | 1.3 GB | 63 | 1m back-adjusted + 1m unadjusted + 5/15/60/240/1D resamples × 4 instruments |
| `data/features/` | 1.7 GB | 33 | 4 timeframes × 4 instruments × (parquet + manifest) plus a few base-v2 files |
| **`data/` total** | **3.7 GB** | **450** | |
| `runs/` | 221 MB | 404 | 19 strategy directories + sweep outputs |
| `outputs/` | <1 MB | (small) | work logs, planning docs |
| **Project total (excl. `.venv/`)** | ~3.9 GB | ~1,700 | source, tests, configs, docs included |

`.venv/` (the Python virtual environment) is large in file count
(~18,600 vendored package files) but constant in size and not part of
the storage management problem — `uv sync` rebuilds it idempotently.
Cleanup policy ignores `.venv/`.

The features layer is the largest absolute consumer — every
per-instrument feature parquet carries the indicator stack inline
rather than referencing CLEAN by computation. This is a deliberate
choice (Chapter 4 §4.1) and not a candidate for optimisation; the
pessimistic-default for storage is "denormalise so reads are fast and
unambiguous."

## 56.5.2 What grows over time

Six categories of files grow during normal operation. Three are
auto-cleanable in principle; three should not be touched by automation.

### 56.5.2.1 Auto-cleanable

| Category | Where | How fast it grows |
|----------|-------|-------------------|
| Old date-stamped CLEAN parquets | `data/clean/{symbol}_*_{YYYY-MM-DD}_{YYYY-MM-DD}.parquet` | One redundant file per `rebuild clean` after a new bulk download. Weekly refresh ⇒ ~50 redundant files/instrument/year |
| Old date-stamped FEATURES parquets | `data/features/{symbol}_*_features_{tag}_*.parquet` | Same cadence as CLEAN, multiplied by the number of feature-set tags in active use |
| Sweep run directories | `runs/{strategy_id}/{YYYY-MM-DD-HH-MM-SS}/` | One per sweep variant. The session-38 96-trial sweep produced ~96 directories, ~150 MB total |

### 56.5.2.2 Should-not-touch

| Category | Where | Why preserved |
|----------|-------|---------------|
| RAW contracts | `data/raw/contracts/` | Re-download is hours and consumes vendor quota; RAW is the audit trail of what TradeStation returned |
| Validation-mode trials | entries in `runs/.trials.json` with `mode="validation"` | These are the gate-passing record; pruning them destroys the audit trail of what was promoted |
| Work logs | `outputs/work-log/` | Project narrative history; small (kilobytes per session) |

### 56.5.2.3 The trial registry specifically

`runs/.trials.json` is append-only and currently holds 169 trial
records (~100 KB). The growth rate is one entry per backtest, walk-
forward, or sweep variant. The registry is the input to deflated-
Sharpe computation (Chapter 23 §23.3), so an aggressive prune of
exploration-mode trials would understate the multiple-testing
correction.

The retention policy in §56.5.4 keeps validation-mode trials forever
and archives exploration-mode trials older than 180 days into a
parallel `outputs/archive/trials/` file rather than deleting them.

## 56.5.3 Cleanup CLI commands  **[GAP]**

This section is the implementation spec for the `clean` subcommand
group. Five subcommands; one safety pattern shared across all of them.

### 56.5.3.1 The shared safety pattern

Every `clean` subcommand obeys the following invariants:

1. **Dry-run is the default.** Without `--apply`, the command prints
   what *would* be deleted, totals the bytes reclaimed, and exits 0
   without touching the filesystem. The `--apply` flag is required to
   actually delete.
2. **Archive before delete.** Unless `--no-archive` is passed, every
   reaped file is first written to `outputs/archive/` under a
   per-category, per-month directory (`runs/<strategy_id>/<YYYY-MM>.tar.gz`,
   `clean/<YYYY-MM>.tar.gz`, etc.). The archive write must succeed
   before the original is deleted; if archiving fails, the delete is
   skipped and the file is logged as `archive_failed`.
3. **Manifest-aware.** A file is *not* a candidate for deletion if its
   path appears as a `sources[].path` entry in any non-deleted
   manifest. The check is performed *after* dry-run summary
   construction; the summary distinguishes "reapable" from
   "manifest-pinned." Without this guard, deleting an old CLEAN file
   would leave a downstream FEATURES manifest pointing into the void.
4. **Verify-clean precondition.** If `uv run trading-research verify`
   reports staleness, `clean --apply` refuses to run with exit code 3.
   The reasoning: a stale state suggests the operator is mid-rebuild,
   and reaping during a rebuild risks deleting a file that's about to
   be re-cited. Override via `--ignore-staleness` (logged loudly).
5. **Output format.** Default: human-readable table to stdout; summary
   line to stderr. With `--json`: a single JSON object on stdout with
   keys `dry_run`, `reapable`, `pinned`, `bytes_reclaimable`, `errors`.
   This is the CLI-as-API contract (Chapter 3 §3.6).
6. **Exit codes.** 0 = success (dry-run produced output, or apply
   succeeded). 1 = unexpected error (I/O failure, malformed manifest).
   2 = bad usage (conflicting flags, unknown strategy id). 3 = refused
   due to verify staleness.
7. **Logging.** Every reap emits a `structlog` event with `event=clean.reap`,
   `action=archive` or `action=delete`, the file path, byte size, and
   the subcommand-specific reason. Log fields are documented in
   Chapter 52 (Logging).

### 56.5.3.2 `clean runs` — reap old run directories

```
trading-research clean runs [--strategy <id>] [--keep-last N | --older-than DURATION]
                            [--apply] [--no-archive] [--json]
```

Reaps timestamped run directories under `runs/{strategy_id}/`. Two
selection modes (mutually exclusive):

- `--keep-last N` (default `N=10`): keep the N most-recent timestamps
  per strategy; reap the rest.
- `--older-than DURATION` (e.g. `90d`, `6m`): reap any run whose
  timestamp is older than the cutoff, regardless of count.

`--strategy <id>` scopes the operation to one strategy directory. Without
it, all strategies are processed.

**Always preserved, never reaped:**

- The single most recent run directory per strategy (even with
  `--older-than`).
- Any run whose `summary.json` carries `mode: "validation"` — these are
  gate runs and are part of the audit trail.

**Examples:**

```
# Preview reaps across all strategies, keeping last 10:
trading-research clean runs

# Reap to a 90-day window for one strategy, applying:
trading-research clean runs --strategy 6a-monthly-vwap-fade-yaml-v2b \
    --older-than 90d --apply

# JSON output for scripting:
trading-research clean runs --json
```

### 56.5.3.3 `clean canonical` — reap old CLEAN parquets

```
trading-research clean canonical [--symbol <SYM>] [--keep-latest]
                                 [--apply] [--no-archive] [--json]
```

For each (symbol, timeframe, adjustment) tuple, identifies the parquet
with the latest end-date suffix and reaps the older variants. The
operation is manifest-aware (§56.5.3.1 invariant 3): a CLEAN parquet
cited by any non-reaped FEATURES manifest's `sources[]` is held back.

The subcommand is named `canonical` because the CLEAN layer is the
canonical OHLCV form (Chapter 4 §4.1) — this avoids the awkward
`clean clean` collision between the subcommand group name and the
layer name.

`--keep-latest` is the default behaviour and is shown explicitly for
readability. There is no other selection mode in v1; a future
`--keep-last N` may be added if multiple historical windows turn out
to be useful.

### 56.5.3.4 `clean features` — reap a feature-set tag

```
trading-research clean features [--tag <tag>] [--symbol <SYM>] [--keep-latest]
                                [--apply] [--no-archive] [--json]
```

Two operating modes:

- **`--tag <tag>` (no `--keep-latest`)**: reap *all* FEATURES files
  for the named feature-set tag. Use this when retiring an experiment
  tag (`experiment-13min`, `base-v0`, etc.). The corresponding YAML in
  `configs/featuresets/` is left in place — git history is the audit
  trail for the config — but the operator may delete it manually.
- **`--keep-latest`**: per (symbol, timeframe, tag) keep only the
  most-recent end-date variant; reap the older ones. Same date-suffix
  reaping as `clean canonical`.

The two modes are mutually exclusive. `--tag` without `--keep-latest`
reaps the entire tag; `--keep-latest` without `--tag` keeps the latest
of every tag.

### 56.5.3.5 `clean trials` — three-tier prune of the trial registry

The trial registry is the platform's audit trail of every variant
ever tested. Pruning it carelessly destroys the multiple-testing
correction (Chapter 33) and the deflated-Sharpe count (Chapter 23).
Pruning it never destroys disk pressure either — the file would have
to grow past tens of megabytes before that mattered. The
discipline is therefore *compaction first, deletion only after a long
quiet period.*

```
trading-research clean trials [--apply] [--json]
                              [--compact-after DURATION]
                              [--delete-after DURATION]
                              [--keep-mode validation]
```

Three tiers, applied in order:

| Tier | Where the entry lives | Default age threshold | What `clean trials` does |
|------|----------------------|----------------------|--------------------------|
| **Live** | `runs/.trials.json` | < 180 days, or `mode == "validation"` (any age) | Nothing |
| **Compacted archive** | `outputs/archive/trials/{YYYY-MM}.jsonl` | ≥ 180 days, exploration-mode | Move from live registry into a monthly compacted JSONL file; remove from live |
| **Deletion** | (gone) | ≥ 730 days (2 years), exploration-mode, *and* not referenced by any live `parent_sweep_id` | Delete from the archive entirely |

Defaults:

- `--compact-after 180d` — exploration trials older than this move to
  the compacted archive on the next `clean trials --apply` run.
- `--delete-after 730d` — exploration trials older than this *and*
  unreferenced by any current sweep are deleted from the archive.
  Validation trials are never deleted at any age.
- `--keep-mode validation` (always on; can be disabled only with the
  explicit override `--prune-validation` which prints a loud warning
  and requires a confirmation flag).

The "referenced by a live `parent_sweep_id`" check is the safety
guard against deleting a trial that a current sweep still names as
its parent — this would orphan the sweep's lineage. The check walks
the live registry once before any deletion.

**The deflation count remains correct in every tier.** The deflated-
Sharpe and multiple-testing modules consult both the live registry
*and* the compacted archive when counting historical trials. A
deletion-tier removal does reduce the historical count, which is
correct — at two years' age, an exploration trial that was never
revisited or referenced is no longer a meaningful contributor to
multiple-testing bias for present work.

Output of dry-run:

```
Trial registry:        169 entries
  ── live (kept):           58
  ── compactable:          103   →  outputs/archive/trials/2025-09.jsonl ... 2026-04.jsonl
  ── deletable:              8   →  (delete from existing archive: 2024-04.jsonl, 2024-05.jsonl)

After apply:           58 live entries, 103 archived, 8 deleted.
Bytes reclaimed:       ~12.4 KB (live) + 1.6 KB (archive)
```

The bytes reclaimed are small in absolute terms but the operation is
about *intelligibility* of the registry as much as disk: a 169-entry
registry is readable; a 5,000-entry registry is not, even if it fits
on disk.

### 56.5.3.6 `clean dryrun` — explicit preview across all categories

```
trading-research clean dryrun [--json]
```

Convenience command equivalent to running `clean runs`, `clean
canonical`, `clean features --keep-latest`, and `clean trials` in
sequence, all in dry-run mode. Produces a single combined summary:

```
Category           Reapable      Pinned      Bytes reclaimable
runs               42 dirs       3 dirs      142.7 MB
canonical          18 files      2 files     284.1 MB
features           5 files       0 files     51.2 MB
trials (compact)   87 entries    —           14.2 KB
trials (delete)    4 entries     —           0.6 KB
──────────────────────────────────────────────────────────────
Total                                         478.0 MB
```

This is the command the operator runs first, every time. `--apply` is
disallowed on `dryrun`; to actually reap, run the per-category
commands.

## 56.5.4 Retention policy (default)  **[GAP]**

The defaults below are what `clean` commands enforce when no
overrides are passed. They balance disk pressure against
audit-trail preservation. The defaults can be tuned per-operator in
`configs/retention.yaml` (also a [GAP]; spec follows).

| Layer | Default policy |
|-------|----------------|
| `data/raw/` | **Never automatically deleted.** Re-download is expensive and RAW is the audit trail of vendor data |
| `data/clean/` | Per (symbol, timeframe, adjustment), keep latest end-dated variant; archive older variants to `outputs/archive/clean/{YYYY-MM}/` |
| `data/features/` | Per (symbol, timeframe, feature-set-tag), keep latest end-dated variant; archive others. Retired tags reaped on operator request |
| `runs/` | Keep last 10 runs per strategy plus every `mode=validation` run; archive runs older than 90 days into `outputs/archive/runs/` |
| `runs/.trials.json` | Three tiers: live registry holds < 180 days + all validation-mode; exploration-mode 180d–730d compacted to `outputs/archive/trials/{YYYY-MM}.jsonl`; exploration-mode > 730 days *and* unreferenced by live `parent_sweep_id` deleted. Validation-mode never deleted at any age. DSR/multiple-testing modules read both live + archive |
| `outputs/work-log/` | **Never automatically deleted.** Project narrative; small |
| `outputs/archive/` | Cold storage; never deleted by automated policy |
| `data/.tmp/` | Empty at end of every successful pipeline run; populated only mid-run |

### 56.5.4.1 The retention YAML  **[GAP]**

`configs/retention.yaml` (planned, not yet present) overrides the
defaults at site level:

```yaml
runs:
  keep_last_per_strategy: 10
  archive_older_than: 90d
  archive_dir: outputs/archive/runs/

clean:
  keep_latest_per_tuple: true
  archive_dir: outputs/archive/clean/

features:
  keep_latest_per_tuple: true
  retire_tags: []                    # explicit list of tags to reap entirely

trials:
  keep_modes: [validation]              # never compacted or deleted at any age
  compact_after: 180d                   # exploration → compacted archive
  delete_after: 730d                    # exploration: archive → fully deleted
  archive_path: outputs/archive/trials/
```

A missing key uses the documented default. A missing file uses all
documented defaults — `clean` does not require the YAML to exist.

## 56.5.5 Intermediate-file inventory  **[EXISTS — to verify]**

The pipeline today writes parquet directly to canonical paths without
intermediate temp files. There are two suspected exceptions to verify
and document in a follow-on revision:

1. **The data downloader** — does TradeStation streaming write to a
   temp file with atomic rename, or does it stream directly to the
   final path? If the former, document the temp location; if the
   latter, document that there is no intermediate state.
2. **The back-adjuster** — does the continuous-contract stitcher
   accumulate in memory, write a scratch parquet, or stream? Memory
   accumulation is fine for current data sizes but should be explicit.

If either turns out to write a temp file, the spec is: temp files
land under `data/.tmp/` with a parent process PID in the filename;
they are reaped at end-of-run by a `try/finally` block; on crash the
operator can delete `data/.tmp/` wholesale without consequence.

## 56.5.6 Growth-rate forecast table

The per-instrument forecast for new instruments is documented in
[Chapter 4 §4.12.3](04-data-pipeline.md#4123-growth-rate-forecast-for-unregistered-instruments).
This section adds the *project-level* forecast: how the operator
should expect total disk usage to grow as a function of activity.

### 56.5.6.1 Forecast under typical operation

Assumptions: 4 instruments fully built, weekly `pipeline` refresh
extending the date range by ~5 trading days each, one feature-set tag
in active use, ~30 backtest runs per week split across ~5 strategy
directories, no cleanup ever run.

| Time horizon | Approximate `data/` | Approximate `runs/` | Total |
|--------------|---------------------|---------------------|-------|
| Today | 3.7 GB | 0.2 GB | 3.9 GB |
| +1 month | 4.2 GB | 0.4 GB | 4.6 GB |
| +3 months | 5.5 GB | 1.1 GB | 6.6 GB |
| +6 months | 7.3 GB | 2.2 GB | 9.5 GB |
| +12 months | 10.8 GB | 4.4 GB | 15.2 GB |

The `data/` growth is dominated by date-stamped CLEAN/FEATURES
duplicates (the rebuild pattern leaves the prior file in place); the
`runs/` growth is dominated by sweep-output directories.

### 56.5.6.2 Forecast under default retention

Same activity profile, with `clean dryrun --apply`-equivalent reaping
run weekly:

| Time horizon | Approximate `data/` | Approximate `runs/` | Total |
|--------------|---------------------|---------------------|-------|
| Today | 3.7 GB | 0.2 GB | 3.9 GB |
| +3 months | 4.0 GB | 0.4 GB | 4.4 GB |
| +12 months | 4.5 GB | 0.6 GB | 5.1 GB |

The slope flattens because each refresh's prior parquet is reaped at
the next cleanup pass; only the actively cited variant remains. The
12-month total is bounded by the per-instrument cost (§56.5.1) plus
the active strategy count.

### 56.5.6.3 Sizing guidance — when to act

Cross-references the table in Chapter 4 §4.12.5; restated here for
proximity:

| Total `data/` size | Action |
|--------------------|--------|
| Under 5 GB | No action needed |
| 5–10 GB | Run `clean dryrun` weekly; apply when reapable totals exceed 1 GB |
| 10–25 GB | Run `clean runs --older-than 90d --apply` and `clean canonical --apply` monthly |
| Over 25 GB | Audit per-instrument footprint via `inventory`; consider archiving cold instruments by reaping their FEATURES tag entirely |

The `status` CLI (Chapter 49.16, **[GAP]**) will surface the current
total and a recommendation in its first paragraph of output.

## 56.5.7 What never gets cleaned

A short list, restated for prominence:

- **RAW contracts** (`data/raw/contracts/`): never auto-reaped. The
  operator may delete one manually if a vendor data correction
  requires re-download, but no `clean` subcommand touches RAW.
- **Manifest sidecars**: only deleted when the parquet they describe
  is deleted. Orphaned manifests (parquet missing, manifest present)
  are *flagged* by `verify` but not reaped — the operator decides.
- **Validation-mode trials** in `runs/.trials.json`: never compacted
  to archive, never deleted, regardless of age. They are the
  gate-passing audit trail.
- **Work logs** (`outputs/work-log/`): never reaped.
- **Cold-storage archives** (`outputs/archive/`): once written, never
  reaped by automated policy. The operator may compress or move these
  manually if the disk pressure on the archive itself becomes a
  concern.
- **Configuration files** (`configs/`): never touched by `clean`.
  Retiring a feature-set tag reaps the FEATURES files; the YAML stays
  in `configs/featuresets/` until the operator explicitly removes it
  and commits the removal.
